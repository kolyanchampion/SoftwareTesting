# 3. НЕФУНКЦІОНАЛЬНЕ ТЕСТУВАННЯ

## 3.1 Теоретичні засади

Нефункціональне тестування оцінює характеристики якості системи, які не є частиною її основної функціональності: продуктивність, надійність, безпека, зручність використання та сумісність.

## 3.2 Тест-кейс 1: Продуктивність

**Тип нефункціонального тестування:** Продуктивність (Performance Testing)

**Ідентифікатор:** NFT-LR5-TC-001

**Назва:** Вимірювання часу завантаження головної сторінки

**Метод тестування:** Automated testing з використанням Google Lighthouse

**Кроки виконання:**

1. Відкрити DevTools у браузері Chrome
2. Перейти на вкладку Lighthouse
3. Вибрати категорії "Performance", "Best Practices"
4. Натиснути "Analyze page load"
5. Дочекатися завершення аналізу
6. Зафіксувати метрики продуктивності

**Метрики продуктивності:**

| Метрика | Значення | Стандарт ISTQB | Результат |
|---------|---------|----------------|-----------|
| First Contentful Paint (FCP) | 0.8 сек | < 1.8 сек | PASS |
| Largest Contentful Paint (LCP) | 1.4 сек | < 2.5 сек | PASS |
| Time to Interactive (TTI) | 1.8 сек | < 3.8 сек | PASS |
| Cumulative Layout Shift (CLS) | 0.05 | < 0.1 | PASS |
| Performance Score | 95 | > 90 | PASS |

**Висновок:** Головна сторінка завантажується з високою продуктивністю, всі метрики відповідають стандартам.

## 3.3 Тест-кейс 2: Безпека

**Тип нефункціонального тестування:** Безпека (Security Testing)

**Ідентифікатор:** NFT-LR5-TC-002

**Назва:** Тестування стійкості до XSS-атак

**Кроки виконання:**

1. Авторизуватися на платформі
2. Відкрити будь-який курс з можливістю залишення коментаря
3. Відкрити DevTools (F12)
4. Знайти поле введення коментаря
5. Ввести тестовий XSS-вектор: `<script>alert('XSS Test')</script>`
6. Зберегти коментар
7. Переглянути збережений коментар на сторінці

**Критерій успішності:** Скрипт не повинен виконатися; система має екранувати HTML-символи

**Фактичний результат:** Скрипт відображається як звичайний текст, виконання заблоковано

**Висновок:** Система має ефективний захист від XSS-атак

## 3.4 Тест-кейс 3: Юзабіліті

**Тип нефункціонального тестування:** Зручність використання (Usability Testing)

**Ідентифікатор:** NFT-LR5-TC-003

**Назва:** Оцінка часу для виконання типової задачі

**Кроки виконання:**

1. Засікти час за допомогою секундоміра
2. Почати з головної сторінки (без авторизації)
3. Знайти курс "Веб-розробка для початківців"
4. Авторизуватися
5. Записатися на курс
6. Зупинити секундомір після появи підтвердження

**Критерій успішності:** Середній час виконання < 60 секунд для досвідченого користувача

**Фактичний результат:** Час виконання склав 42 секунди

**Висновок:** Процес є інтуїтивним та ефективним

# 5. АНАЛІЗ РЕЗУЛЬТАТІВ ТЕСТУВАННЯ

## 5.1 Відповіді на контрольні питання

### Питання 1: Чому нефункціональне тестування є критично важливим?

Нефункціональне тестування є критично важливим з кількох фундаментальних причин.

По-перше, користувачі формують враження про продукт протягом перших секунд взаємодії. Дослідження показують, що 53% мобільних користувачів залишають сайт, якщо він завантажується більше трьох секунд. Кожна секунда затримки знижує конверсію та збільшує показник відмов.

По-друге, безпекові вразливості можуть призвести до катастрофічних наслідків: витоку персональних даних користувачів, фінансових втрат та непоправної репутаційної шкоди. Регулярні скандали з витоком даних великих компаній підтверджують важливість систематичного тестування безпеки.

По-третє, сумісність з різними пристроями, браузерами та операційними системами визначає охоплення потенційної аудиторії. Платформа, що працює лише в одному браузері, втрачає значну частину користувачів.

По-четверте, надійність та доступність системи впливають на операційні витрати бізнесу. Система, що часто падає, вимагає більше ресурсів на підтримку та завдає шкоди бізнесу клієнтів.

### Питання 2: Чи можна вважати продукт якісним, якщо він працює правильно, але повільно?

Ні, продукт не можна вважати повністю якісним у такому випадку. Якість програмного забезпечення за міжнародними стандартами ISO 25010 включає вісім комплексних характеристик: функціональну придатність, продуктивність та швидкодію, сумісність, зручність використання, надійність, безпеку, підтримуваність та мобільність.

Швидкодія є невід'ємною частиною користувацького досвіду. Повільна система може фізично працювати правильно, але завдавати користувачам значних незручностей. У корпоративному середовищі повільна система знижує продуктивність працівників, що має прямий фінансовий вплив на організацію.

Для багатьох типів систем швидкодія є критичною вимогою. Системи реального часу, торгові платформи, фінансові застосунки не можуть функціонувати належним чином при недостатній продуктивності.

Тому повноцінна якість вимагає відповідності всім характеристикам стандарту, включаючи продуктивність та швидкодію.

### Питання 3: Який тип тестування, на вашу думку, складніше автоматизувати і чому?

На мою думку, юзабіліті тестування є найскладнішим для автоматизації.

Головна проблема полягає у фундаментальній суб'єктивності людського сприйняття. Автоматизовані інструменти можуть виміряти час виконання задачі, кількість кліків або відсоток успішних дій, але не можуть оцінити емоційну реакцію користувача, рівень його розчарування або ступінь інтуїтивності інтерфейсу.

Тестування доступності (accessibility) також потребує значної частки ручної роботи. Хоча існують автоматичні сканери, такі як axe-core або Accessibility Insights, вони виявляють лише частину проблем, особливо ті, що стосуються когнітивної доступності або контекстного розуміння контенту.

Водночас, функціональне тестування є відносно простим для автоматизації завдяки чітким вхідним та вихідним даним. Регресійне тестування добре піддається автоматизації, що дозволяє швидко перевіряти велику кількість сценаріїв після кожної зміни коду.

Компромісним рішенням є використання автоматизації для рутинних перевірок продуктивності, безпеки та сумісності з подальшою ручною оцінкою суб'єктивних факторів користувацького досвіду.

# 6. ВИСНОВКИ

Проведена лабораторна робота дозволила на практиці порівняти функціональне та нефункціональне тестування.

Функціональне тестування зосереджене на перевірці того, що система робить: реєстрація працює коректно, пошук знаходить курси, оплата відкриває доступ до контенту. Ці тести базуються на чітких вимогах та мають однозначний результат.

Нефункціональне тестування перевіряє те, як система працює: швидкість, безпека, зручність використання. Хоча ці характеристики часто є вирішальними для успіху продукту на ринку, їх складніше формалізувати та виміряти.

Класифікація тестових сценаріїв продемонструвала, що більшість стандартних сценаріїв можна однозначно віднести до функціональних або нефункціональних, хоча існують сценарії на межі двох категорій.

Аналіз показав, що нефункціональне тестування є критично важливим для успіху продукту. Продукт може бути функціонально правильним, але непридатним для використання через погану продуктивність або незручний інтерфейс.

Для досягнення найкращих результатів рекомендується планувати обидва типи тестування з самого початку проекту, виділяючи достатні ресурси на кожен із них.
